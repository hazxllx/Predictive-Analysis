Pulse Healthcare / Predictive Assessment System (Admin Sync Notes)

1) What the system does
- Patients run a predictive health assessment (risk scoring + recommendations).
- The system stores each assessment result in MongoDB.
- The admin dashboard shows:
  - aggregated system stats
  - recent assessments
  - a patient registry with each patient’s latest risk status
- Admin views must reflect the patient’s latest saved assessment immediately after re-running.

2) Frontend overview (React)
- Routing / Pages
  - /my-dashboard (patient side)
  - /admin-dashboard (admin overview)
  - /admin/users
  - /admin/patients (admin patient registry)
  - /admin/audit-log
  - /admin/patients/:patientId (patient detail view in admin context)
- Key frontend utilities
  - frontend/src/api/axios.js
    - shared Axios client for API requests
  - frontend/src/api/cachedFetch.js
    - cached fetch helper (queryCache-backed)
  - frontend/src/api/queryCache.js
    - cache storage + invalidation helpers for safe refetch
  - frontend/src/utils/normalizeAssessment.js
    - normalizes raw assessment payloads so UI can safely read:
      risk_level, risk_score, health score, recommendations, timestamps, etc.
  - frontend/src/utils/normalizePatients.js
    - normalizes patient objects coming from PMS/fallback sources
  - frontend/src/utils/recommendationEngine.js
    - derives guidance (used in patient dashboards / UI)
  - frontend/src/pages/PatientDetail.jsx
    - runs/refreshes assessments and updates UI state
  - frontend/src/pages/Patients.jsx
    - admin registry list; shows latest assessment per patient
  - frontend/src/pages/AdminDashboard.jsx
    - admin landing; shows recent assessments feed

3) Backend overview (Node/Express + MongoDB)
- server setup
  - backend/server.js
    - creates Express app and starts server
  - backend/config/db.js
    - MongoDB connection
- Authentication / middleware
  - backend/middleware/auth.js
    - protects admin + patient routes
  - backend/middleware/rateLimiter.js, requestLogger.js
    - stability/observability
- Assessment APIs
  - backend/routes/assessment.js
    - assessment endpoints (create/run, history, latest-by-patient, etc.)
  - backend/controllers/assessmentController.js
    - request handlers for assessment creation and retrieval
  - backend/services/scoring.js and/or backend/services/scoringService.js
    - scoring pipeline (inputs -> risk score/level, recommendations)
  - backend/models/Assessment.js
    - MongoDB schema for stored assessments

4) Important MongoDB collections/models
- backend/models/Assessment.js
  - Stores patient risk assessment results including:
    - risk score (numeric)
    - risk level classification (Critical/High/Moderate/Low)
    - recommendations + insights
    - links to patient identity (userId and patient_id)
    - timestamps (createdAt; may also store timestamp field)
  - Indexes:
    - patient_id + createdAt (chronological retrieval)
    - additional indexes for userId/risk_level lookups

5) Assessment flow (end-to-end)
A) Patient runs an assessment (frontend)
- frontend/src/pages/PatientDetail.jsx
  - user submits/executes assessment
  - frontend sends assessment inputs to backend run/create endpoint
  - after success, UI updates:
    - patient page state
    - local history list (append latest assessment)
  - critical requirement: admin views must refresh so they show the updated latest assessment

B) Backend scoring + persistence
- backend/controllers/assessmentController.js (and related services)
  - validates inputs
  - computes risk score + risk level
  - builds recommendations and insights
  - saves an Assessment document
  - ensures the “latest assessment” endpoints return the newest document (by createdAt/timestamp)

C) Admin synchronization (frontend cache + refetch)
- Admin pages depend on “latest-by-patient” and “recent assessments” data.
- To avoid stale displays:
  - PatientDetail invalidates cached assessment queries after a successful run.
  - Admin pages force-refetch/refresh when reopened or brought back to focus.

6) How admin synchronization works (the stabilization implemented)
Key goal: prevent stale/old assessment values from persisting in admin pages.

A) PatientDetail -> cache invalidation after assessment run
- frontend/src/pages/PatientDetail.jsx
  - after successful assessment persistence:
    - invalidates cached keys used by admin pages:
      - ["assessments","latest-by-patient"]
      - ["assessments","all"] (or related recent assessments)
    - invalidates other patient-level caches as needed
  - this forces admin pages to request the latest data from the backend

B) Admin Patients registry -> refresh on tab focus
- frontend/src/pages/Patients.jsx
  - loads latest-by-patient once on mount
  - additionally, when user.role === "admin":
    - listens to window "focus"
    - forces a refresh of latest-by-patient and rebuilds the in-memory assessments map
  - this ensures “rerun assessment” updates appear immediately when the admin returns

C) Admin Dashboard -> bypass cached “recent assessments”
- frontend/src/pages/AdminDashboard.jsx
  - the “recent assessments” feed fetch is forced with caching bypass:
    - fetchWithCache({ ..., force: true })
  - ensures updated risk score / risk level / timestamps are shown on the dashboard quickly

7) How analytics/charts compute values
- Admin and patient charts rely on “latest assessment” data:
  - risk_score, risk_level, health score (normalized in normalizeAssessment.js)
  - recommendation lists
- Frontend components (examples)
  - frontend/src/components/RadialHealthScore.jsx
  - frontend/src/components/ScoreBreakdown.jsx
  - frontend/src/components/ResultTabs.jsx
- The normalization utilities are the single point that converts backend fields
  into what charts expect, reducing inconsistent rendering.

8) Authentication flow
- Backend
  - backend/routes/auth.js and backend/controllers/authController.js
  - backend/middleware/auth.js validates user sessions/tokens
- Frontend
  - frontend/src/context/AuthContext.jsx
    - stores current user role (admin vs patient)
    - chooses role home routes (admin-dashboard vs my-dashboard)
- Admin routes are protected and require admin role.

9) Security/validation handling (practical notes)
- Use backend input validation middleware where present (backend/middleware/validate.js).
- Ensure endpoints only expose patient assessments to authenticated admins/patients.
- For reliability:
  - ensure scoring functions clamp values to safe ranges (0–100) and handle missing inputs.
  - backend should handle “no assessment yet” gracefully.

10) PMS AND ADMIN SUBSYSTEM INTEGRATION
Dedicated section: PMS <-> Assessment <-> Admin

A) How the system connects with the PMS (Patient Management System)
- Patients data is fetched from a PMS integration service.
- If PMS is temporarily unavailable, the system can fall back to cached/stored fallback patient records.
- Frontend shows which source is used via patientResponse?.source.

B) How patient records are shared/synchronized with the PMS
- The system’s patient identifiers are aligned via:
  - patient.patient_id (PMS patient ID)
  - and/or user linkage fields (patientLinkingService)
- Backend linking:
  - backend/services/patientLinkingService.js
    - resolves/maintains association between application users and PMS patient records
- This alignment ensures assessments can be stored and retrieved consistently by patient_id.

C) What patient information is retrieved or updated
- Admin registry relies on:
  - patient demographic/lifestyle fields
  - last_visit_date
  - conditions derived from medical normalization logic
- Assessment results rely on:
  - patient_id linkage to the Assessment collection

D) How the assessment module communicates with PMS-related data
- The assessment scoring pipeline uses:
  - patient inputs (from patient-side assessment flow)
  - derived tags from patient conditions/lifestyle
- Storage uses patient_id so PMS-backed patient records can be enriched in admin UI.

E) How admins access synchronized patient assessment data
- Admin pages depend on “latest-by-patient” endpoints and assessment lists:
  - /api/v1/predictive-analysis/risk-assessment/latest-by-patient
  - /api/v1/predictive-analysis/risk-assessment/all?limit=...
- Admin pages map latest assessment results to each patient record:
  - frontend/src/pages/Patients.jsx builds an assessments map keyed by patient_id.
  - frontend/src/pages/AdminDashboard.jsx merges recent assessments with patient names.

F) How analytics/statistics are generated from synchronized patient records
- Admin stats cards (/admin/stats) and feeds
  - use backend aggregation + live assessment documents
- Charts/components use normalized assessment fields:
  - risk_score, risk_level, recommendations, timestamps

G) How admin subsystem avoids stale data issues
- The main stale-data issue is cache-driven UI persistence.
- Fix approach in this system:
  - invalidate/force refresh cached assessment queries immediately after assessment run
  - refresh admin lists when returning/focusing admin pages
  - bypass cache for “recent assessments” on admin dashboard load

H) Files/controllers/services involved (implementation-connected)
- PMS-facing:
  - backend/services/pmsService.js
  - backend/services/pmsData.js
  - backend/controllers/patientController.js and backend/routes/patients.js
  - backend/services/patientLinkingService.js
- Assessment:
  - backend/routes/assessment.js
  - backend/controllers/assessmentController.js
  - backend/services/scoring.js / scoringService.js
  - backend/models/Assessment.js
- Admin/aggregation:
  - backend/routes/admin.js (admin stats)
- Frontend sync:
  - frontend/src/pages/PatientDetail.jsx (invalidation after assessment)
  - frontend/src/pages/Patients.jsx (force refresh on focus for admin registry)
  - frontend/src/pages/AdminDashboard.jsx (force refresh for recent assessments feed)

11) Important architectural notes (no major redesign)
- This system is intentionally incremental:
  - preserve existing UI/routes/db structure
  - focus on safe synchronization and cache correctness
- The stabilization mainly targets:
  - cache invalidation/force refresh timing
  - normalization consistency (normalizeAssessment.js)
  - ensuring the “latest assessment” endpoints correspond to the newest Assessment document.
