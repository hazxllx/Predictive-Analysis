====================================================
1. SYSTEM OVERVIEW
====================================================

Pulse Prophet is a production-grade healthcare Predictive Analysis System integrated with a Patient Management System (PMS). It performs:

- Rule-based risk scoring using medically validated taxonomy
- Patient assessment with comprehensive risk breakdown
- Recommendation generation based on risk factors
- Specialist and laboratory test suggestions
- Assessment history tracking
- Role-based access control (admin/patient)

The system is designed to:
- Never accept arbitrary free-text as medical diagnoses
- Normalize all clinical inputs to canonical labels
- Provide clinically meaningful risk assessments
- Support Filipino/common Philippine health risks
- Be production-ready with security hardening

6. SCORING ENGINE
====================================================

Location: `backend/utils/scoringEngine.js`

The scoring engine is a balanced, preventive wellness calculator that recognizes BOTH risk factors and healthy behaviors. It was refactored to prevent over-reaction and produce realistic, non-alarmist scores.

Philosophy:
- Low = genuinely healthy
- Moderate = common/manageable risks (the NORMAL outcome)
- High = multiple concerning risks
- Critical = rare severe combinations only

Why the previous system over-reacted:
1. Condition weights were too high (e.g., Hypertension = 25, Diabetes = 30)
2. Multiple cardiovascular conditions stacked without limit
3. No protective/negative scoring for healthy behaviors
4. Escalation bonuses were large (+12 to +15)
5. Risk thresholds were too low (Critical at 70)
6. No distinction between controlled and uncontrolled conditions
7. Dead code in scoring.js added duplicate bonuses on top of the engine

New Scoring Formula:
FINAL SCORE = (Risk Factors) - (Protective Factors) + (Escalation Adjustments)
Then: apply category caps → normalize → clamp 0–100

Components:

getAgePoints(age)
- 65+ years = 12 points
- 50–64 years = 8 points
- 40–49 years = 5 points
- <40 years = 0 points

Condition Weights
- All registry weights are scaled by 0.5 to prevent over-reaction
- Controlled conditions receive 50% reduction (e.g., well-managed Hypertension = 6 pts)
- Uncontrolled conditions receive full scaled weight

getVitalsContributions(vitals)
- Blood Pressure:
  - ≥160/100 = 6 points (Severe hypertension)
  - 140-159/90-99 = 5 points (Stage 2)
  - 130-139/85-89 = 3 points (Elevated)
  - <120/80 = -8 points (Normal — PROTECTIVE)
- Heart Rate:
  - >100 = 3 points (Tachycardia)
  - <50 = 2 points (Bradycardia)
  - 60-100 = -5 points (Healthy — PROTECTIVE)
- Respiratory Rate:
  - Abnormal = 2 points
  - 12-20 = -4 points (Normal — PROTECTIVE)
- Temperature:
  - ≥100.4°F = 2 points
- BMI:
  - ≥35 = 5 points (Class 2 obesity)
  - 30-34.9 = 4 points (Obesity)
  - 25-29.9 = 2 points (Overweight)
  - 18.5-24.9 = -6 points (Healthy — PROTECTIVE)

getProtectiveFactors(lifestyle, vitals, lastVisitDate)
- Balanced Diet = -8
- Active Lifestyle = -10
- Light Physical Activity = -4
- Non-Smoker = -8
- No Alcohol Use = -5
- Good Sleep Habits = -4
- Low Stress Level = -3
- Normal Blood Pressure = -8 (also in vitals)
- Healthy Heart Rate = -5 (also in vitals)
- Normal Respiratory Rate = -4 (also in vitals)
- Healthy BMI = -6 (also in vitals)
- Routine Checkups (≤6 months) = -4
- Good Medication Adherence = -5

isConditionControlled(tag, vitals, lifestyle)
- Detects controlled vs uncontrolled status using vitals + medication adherence
- Supported: Hypertension, Diabetes, Type 1 Diabetes, Type 2 Diabetes, Asthma, COPD
- Well-managed → 50% weight reduction
- Uncontrolled → full weight
- Unknown → full weight

applyCategoryCaps(breakdown)
- Prevents overlapping diseases from stacking infinitely
- Cardiovascular cap = 30
- Metabolic cap = 25
- Respiratory cap = 20
- Renal cap = 25
- Neurological cap = 20
- Mental Health cap = 15
- Oncology cap = 25
- Reproductive cap = 15
- Infectious cap = 15
- Musculoskeletal cap = 15
- Lifestyle cap = 18
- Family History cap = 15
- Excess points above cap are shown as negative "cap applied" adjustments

applyRiskEscalation(canonicalTags, breakdown)
- Reduced bonuses for dangerous combinations
- Examples:
  - Hypertension + Diabetes + Smoking = +6
  - CKD + Diabetes = +7
  - Obesity + Sedentary Lifestyle = +4
  - Smoking + COPD/Asthma = +5
  - Stroke + Heart Disease = +7
  - Hypertension + High Cholesterol = +4
  - Diabetes + High Cholesterol = +4
  - Pregnancy + Hypertension = +5
  - Pregnancy + Diabetes = +4
  - ≥3 Family History risks = +3
  - ≥4 Lifestyle risks = +3

buildRecommendations(canonicalTags, riskLevel)
- Uses taxonomy-based recommendations per condition
- Adds risk-level specific recommendations (preventive, not alarmist)
- Low risk: "Continue your healthy habits", "Schedule an annual wellness check-up"
- Critical: "Schedule a comprehensive health assessment as soon as possible"
- Universal recommendations always included
- Returns up to 14 recommendations

buildSpecialists(canonicalTags, riskLevel, vitals)
- Uses taxonomy-based specialists per condition
- Adds Cardiologist if BP elevated
- Default to Internal Medicine if High/Critical and no specialists

buildLabTests(canonicalTags, riskLevel, vitals)
- Uses taxonomy-based lab tests per condition
- Adds BP Monitoring + ECG if BP elevated
- Adds comprehensive labs if High/Critical

scorePatient(data)
Main scoring function:
1. Extract and normalize clinical tags from patient data
2. Add age points (reduced)
3. Add per-condition weights from taxonomy (scaled ×0.5)
4. Apply controlled/uncontrolled adjustments
5. Add vitals contributions (risk + protective)
6. Add protective factors from lifestyle
7. Apply risk escalation bonuses (reduced)
8. Apply category caps
9. Clamp final score to 0-100
10. Determine risk level (new thresholds)
11. Generate recommendations, specialists, lab tests
12. Return complete assessment object

Output Structure:
{
  patient_id: string,
  riskScore: number (0-100),
  risk_level: "Low" | "Moderate" | "High" | "Critical",
  recommendations: string[],
  specialists: string[],
  lab_tests: string[],
  breakdown: [{ label, category, points }],  // includes negative values
  score_breakdown: { [label]: points },
  disclaimer: string,
  clinical_tags: string[] (canonical labels)
}

====================================================
7. RISK ESCALATION LOGIC
====================================================

Risk escalation rules detect dangerous condition combinations and add small bonus points. These are now REDUCED to prevent over-reaction.

Major Escalations (Reduced):

Cardiovascular Crisis:
- Hypertension + Diabetes + Smoking → +6 points (was +12)
- Reason: Triple threat for heart disease and stroke

Renal-Metabolic Crisis:
- CKD + Diabetes → +7 points (was +15)
- Reason: Diabetes accelerates kidney damage

Metabolic Compounding:
- Obesity + Sedentary Lifestyle → +4 points (was +8)
- Reason: Synergistic metabolic risk

Respiratory Crisis:
- Smoking + COPD or Asthma → +5 points (was +10)
- Reason: Smoking exacerbates respiratory disease

Neuro-Cardio Crisis:
- Stroke + Heart Disease → +7 points (was +14)
- Reason: Shared vascular pathology

Atherosclerosis Acceleration:
- Hypertension + High Cholesterol → +4 points (was +8)
- Reason: Accelerated plaque buildup

Dyslipidemia-Diabetes:
- Diabetes + High Cholesterol → +4 points (was +8)
- Reason: Multi-factor cardiovascular risk

Pregnancy Complications:
- Pregnancy + Hypertension → +5 points (Preeclampsia risk) (was +10)
- Pregnancy + Diabetes → +4 points (Gestational diabetes) (was +8)

Genetic Burden:
- ≥3 Family History conditions → +3 points (was +6)
- Reason: Significant genetic risk

Lifestyle Clustering:
- ≥4 Lifestyle risk factors → +3 points (was +6)
- Reason: Behavioral compounding

10. API SPECIFICATION
====================================================

All endpoints return consistent response format:

Success:
{
  "success": true,
  "message": string (optional),
  "data": object/array,
  ...
}

Error:
{
  "success": false,
  "message": string
}

Authentication Endpoints:

POST /auth/register
Request: { identifier, password, name, email }
Response: { success, message, token, user }

POST /auth/login
Request: { identifier, password }
Response: { success, message, token, user, patientLink }

GET /auth/me
Headers: Authorization: Bearer <token>
Response: { success, data: { user } }

PATCH /auth/me
Headers: Authorization: Bearer <token>
Request: { name, email }
Response: { success, message, user }

POST /auth/change-password
Headers: Authorization: Bearer <token>
Request: { currentPassword, newPassword }
Response: { success, message }

Assessment Endpoints:

POST /assessment/risk-assessment
Headers: Authorization: Bearer <token>
Request: { patient_id, lifestyle, vitals, condition_categories }
Response: { success, data: { assessment } }

GET /assessment/risk-assessment?id=<patient_id>

====================================================
12. SECURITY HARDENING
====================================================

Implemented Security Measures:

1. Password Security
- bcrypt hashing with cost factor 12
- Strong password policy (8+ chars, upper, lower, number, special)
- Common password blacklist
- Password strength validation

2. Authentication Security
- JWT tokens with 7-day expiration
- Secure token storage in localStorage
- Token injection via axios interceptors
- Automatic logout on 401 responses

3. Rate Limiting
- Login: 5 attempts per 15 minutes per IP
- Register: 3 attempts per hour per IP
- Change Password: 3 attempts per hour per IP
- General API: 100 requests per 15 minutes per IP

4. Input Validation
- express-validator for all inputs
- Sanitization of all user inputs
- NoSQL injection prevention via Mongoose
- XSS prevention via input sanitization

5. Security Headers
- Helmet middleware for security headers
- CORS configuration
- Content Security Policy

6. Error Handling
- Centralized error handler
- No stack traces in production
- Sanitized error messages
- Structured logging

7. API Key Security
- API key validation for external services
- API keys stored in environment variables
- Never exposed to frontend

8. Development Bypass
- Auth dev bypass requires explicit AUTH_DEV_BYPASS=true
- Never enabled in production

13. AUTHENTICATION & AUTHORIZATION
====================================================

Authentication Flow:

1. User Registration:
   - Input validation (name, email, username, password)
   - Email/username uniqueness check
   - Password hashing with bcrypt
   - User creation in MongoDB
   - JWT token generation
   - Audit logging

2. User Login:
   - Accepts identifier (email or username)
   - User lookup by email OR username
   - Password verification with bcrypt.compare()
   - JWT token generation (expires 7 days)
   - Token storage in localStorage
   - Audit logging

3. Session Restoration:
   - GET /auth/me with Bearer token
   - JWT verification
   - User lookup
   - Return user data

4. Password Change:
   - Current password verification
   - New password validation
   - Password hashing
   - Update user document
   - Audit logging

Authorization Roles:

Admin:
- Full access to all endpoints
- Can create/manage users
- Can view all patients
- Can access dashboard statistics

Patient:
- Can only access own patient data
- Can create assessments for self
- Can view own assessment history
- Limited to patient-specific endpoints

====================================================
21. PMS ↔ PAS INTEGRATION ARCHITECTURE
====================================================

Overview:
The Predictive Analysis System (PAS) provides a production-grade REST API for the Patient Management System (PMS) to generate, retrieve, and update patient risk assessments.

Integration Flow:
1. PMS sends patient data to PAS via API
2. PAS validates and normalizes medical conditions
3. PAS generates predictive assessment using scoring engine
4. PAS stores assessment in MongoDB
5. PAS returns assessment to PMS
6. PMS can retrieve assessments anytime
7. PMS can update lifestyle data and regenerate assessments

Authentication:
- PMS endpoints use API key authentication (x-pms-api-key header)
- Internal app endpoints use JWT authentication (Bearer token)
- API key stored in PMS_API_KEY environment variable
- Unauthorized requests return 401/403 status codes

Components:

- Risk escalation (condition combinations)
- Recommendation generation
- Specialist mapping
- Lab test suggestions

- Hypertension + Diabetes + Smoking → +12

- Obesity + Sedentary Lifestyle → +8
- Smoking + COPD/Asthma → +10
- Stroke + Heart Disease → +14
- Hypertension + High Cholesterol → +8
- Diabetes + High Cholesterol → +8
- Pregnancy + Hypertension → +10
- ≥3 Family History risks → +6
- ≥4 Lifestyle risks → +6

Risk Level Thresholds:
- 0-19: Low Risk
- 20-44: Moderate Risk
- 45-69: High Risk
- 70-100: Critical Risk

Specialist Mapping:
- Cardiovascular → Cardiologist

- Respiratory → Pulmonologist
- Renal → Nephrologist
- Mental Health → Psychiatrist, Psychologist
- Cancer → Oncologist

Lab Test Mapping:
- Cardiovascular → ECG, Lipid Profile, BP Monitoring
- Metabolic → HbA1c, Blood Glucose, Insulin Test
- Respiratory → Chest X-ray, Pulmonary Function Test
- Renal → Kidney Function Test, Creatinine Test, Urinalysis

- High BP → BP Monitoring, ECG

Recommendation Categories:
- Critical priority (hypertension, obesity)

- Alcohol moderation (risk-weighted)
- Diet improvements (pattern-based)
- Physical activity (progressive)
- Condition-specific (per medical condition)
- Risk-level specific (per overall score)
- Universal (always included)

ERROR HANDLING
====================================================

HTTP Status Codes:
- 200: Success
- 201: Created
- 400: Bad Request (missing required fields)
- 401: Unauthorized (missing/invalid API key or token)
- 403: Forbidden (insufficient permissions)
- 404: Not Found (patient/assessment not found)
- 422: Unprocessable Entity (validation failed)
- 500: Internal Server Error
- 503: Service Unavailable (health check failure)

Error Response Format:
{
  "success": false,
  "message": "Human-readable error message",
  "missing_fields": [...] // For validation errors (422)
}

Common Errors:
- Missing patient_id: "Missing patient_id"
- Invalid API key: "Forbidden: invalid API key"
- Missing API key: "Unauthorized: missing API key"
- Patient not found: "Patient not found"
- Validation failed: "Invalid request payload" with missing_fields
- Assessment not found: "Assessment not found"
